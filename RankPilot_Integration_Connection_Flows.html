🛠 Developer Guide — Google Search Console

What is OAuth 2.0?

OAuth 2.0 is a secure way to let a user give RankPilot permission to access their Google data WITHOUT ever sharing their Google password. The user logs into Google directly (on Google's servers), approves specific permissions, and Google gives RankPilot a special "access token" — like a temporary key.

Step 1 → Step 2: Initiate OAuth Flow

1
When user clicks "Sign in with Google"

Redirect the user's browser to this Google URL (build it on your backend, not frontend — client_secret must stay server-side):

const authUrl = new URL('https://accounts.google.com/o/oauth2/v2/auth');
authUrl.searchParams.set('client_id', GOOGLE_CLIENT_ID);
authUrl.searchParams.set('redirect_uri', 
  'https://app.rankpilot.com.au/api/auth/gsc/callback');
authUrl.searchParams.set('response_type', 'code');
authUrl.searchParams.set('scope', [
  'https://www.googleapis.com/auth/webmasters.readonly',
  // webmasters = Search Console API scope
].join(' '));
authUrl.searchParams.set('access_type', 'offline');
// 'offline' = get refresh_token (stays logged in forever)
authUrl.searchParams.set('state', clientId); 
// state = your internal client ID for security

res.redirect(authUrl.toString());

2
Google redirects back to your callback URL with a code

URL will look like: https://app.rankpilot.com.au/api/auth/gsc/callback?code=4/ABC123&state=client-uuid

How to Auto-Refresh Expired Tokens

Access tokens expire every hour. Before any GSC API call, check if the token has expired. If so, use the refresh_token to get a new access_token automatically — no user action needed.

async function getGscAccessToken(clientId) {
  const integration = await getIntegration(clientId, 'gsc');
  
  // Check if access token still valid
  if (integration.token_expires_at > new Date()) {
    return decrypt(integration.access_token);
  }
  
  // Token expired — refresh it:
  const res = await fetch('https://oauth2.googleapis.com/token', {
    method: 'POST',
    body: new URLSearchParams({
      refresh_token: decrypt(integration.refresh_token),
      client_id: GOOGLE_CLIENT_ID,
      client_secret: GOOGLE_CLIENT_SECRET,
      grant_type: 'refresh_token'
    })
  });
  const newTokens = await res.json();
  
  // Update stored token
  await updateToken(clientId, 'gsc', newTokens);
  return newTokens.access_token;
}

🛠 Developer Guide — WordPress REST API

Application Passwords vs OAuth

WordPress uses its own simpler authentication system called "Application Passwords". These are 24-character passwords that only work for API calls (not for logging into WordPress). They are much simpler than OAuth — no redirects, no callbacks, just store username + password in your database.

Step 1 → 2: Detect WordPress Site

// When user enters their URL, verify it's a WordPress site:
GET {siteUrl}/wp-json/wp/v2/
Headers: none needed

→ Returns WordPress REST API info if it's a WP site
→ Returns 404 if not WordPress or if REST API is disabled

// Check the response contains:
{ "namespace": "wp/v2", 
  "routes": { "/wp/v2/pages": {...} } }
→ Confirm: yes, this is WordPress ✓

Step 3: Test the Application Password

// Verify credentials with a simple read request:
GET {siteUrl}/wp-json/wp/v2/users/me
Headers:
  Authorization: Basic base64("username:app_password")
  // IMPORTANT: base64 encode "username:app_password"
  // NOT the WordPress login password — the APP password

→ Returns current user info if correct
→ Returns 401 Unauthorized if wrong credentials

// JavaScript:
const auth = btoa(`${username}:${appPassword}`);
const res = await fetch(`${siteUrl}/wp-json/wp/v2/users/me`, {
  headers: { 'Authorization': `Basic ${auth}` }
});

Check Permissions

// Run these test calls to verify capabilities:
GET {siteUrl}/wp-json/wp/v2/pages?per_page=1
// If 200 → can read pages ✓

POST {siteUrl}/wp-json/wp/v2/pages
{ "title": "RankPilot Test", "status": "draft", 
  "content": "Connection test - delete this" }
// If 201 Created → can write pages ✓
// Save the returned "id" to delete the test page

DELETE {siteUrl}/wp-json/wp/v2/pages/{testId}?force=true
// Clean up the test draft

Check for Yoast SEO Plugin

// Yoast SEO adds extra meta fields to the API
GET {siteUrl}/wp-json/wp/v2/pages?_fields=yoast_head_json
// If yoast_head_json appears in response → Yoast detected
// This means we can set meta title + description via API

// Without Yoast, we inject meta tags into page content HTML directly
